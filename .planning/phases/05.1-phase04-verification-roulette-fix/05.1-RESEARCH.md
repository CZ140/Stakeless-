### Pattern 1: Zustand localStorage Init + Persist (established project pattern)

**What:** Read from localStorage on store creation; write to localStorage inside the action setter before calling `set()`.

**When to use:** Any state field that must survive page refresh.

**Source evidence — plinkoStore.ts (HIGH confidence, direct code inspection):**

// Persist: write localStorage inside action

### Anti-Patterns to Avoid

- **Inventing localStorage keys:** Do not use `lastBet_roulette` for selectedChip — the locked decision mandates `selectedChip_roulette` specifically because chip denomination is not a raw bet amount.

- **Modifying ChipRack or RoulettePage:** The bug is entirely in the store. `ChipRack.tsx` already calls `setSelectedChip(amount)` for each chip button click — once the store's setter persists, all chip selections are automatically saved.

## Common Pitfalls

### Pitfall 1: Forgetting to normalize `Number()` on localStorage read

**What goes wrong:** `localStorage.getItem()` returns a string or null. If null is passed to `Number()`, it returns `0`. The `|| 10` fallback handles this, but only if `Number(null) || 10` is understood as `0 || 10 = 10`.

**Why it happens:** Developers assume `localStorage.getItem()` returns a number.

**How to avoid:** Always use `Number(localStorage.getItem('key')) || defaultValue`. The `Number()` wrapper converts null to 0 and string numbers to numbers. The `|| defaultValue` handles both null (0 is falsy) and 0 (explicitly invalid bet).

**Warning signs:** TypeScript type error if you skip `Number()` — `localStorage.getItem()` returns `string | null`, not `number`.

### Pitfall 2: Stale localStorage key after mute key fix

**What goes wrong:** Users who previously had `localStorage.mute = 'true'` set will lose their mute preference after the key is renamed to `isMuted_roulette`. On first load after the fix, `isMuted_roulette` won't exist, so `isMuted` will default to `false` (sound on), even if they had muted it before.

**Why it happens:** Renaming localStorage keys is a one-time migration event; old keys persist forever.

**How to avoid:** This is an acceptable UX tradeoff for v1 — a single one-time reset of mute preference. Do NOT implement a migration/fallback; that adds complexity for a trivial preference. Per the locked decision, simply use `'isMuted_roulette'` directly.

**Warning signs:** Considered a problem only if the old `'mute'` key was broadly shared across multiple concerns — confirmed it was roulette-only.

### Pitfall 3: Writing VERIFICATION.md without verifying file/line references

**What goes wrong:** VERIFICATION.md cites `rouletteStore.ts line 49` but after the bug fix, `selectedChip` init will be at a different line.

**Why it happens:** Bug fix changes line numbers before documentation is written.

**How to avoid:** Write the VERIFICATION.md AFTER applying the code fix. This ensures all cited line numbers reflect the actual post-fix file state.

### Pitfall 4: ROUL-05 and ROUL-06 evidence visibility

**What goes wrong:** ROUL-05 (Dozens) and ROUL-06 (Columns) are listed in requirements but the planner may not list them if they weren't in the original phase objective statement (which mentioned ROUL-01..04 and ROUL-07).

**Why it happens:** The objective stated GINF-06..09, ROUL-01..04, ROUL-07. ROUL-05 and ROUL-06 appear in the REQUIREMENTS.md traceability table as Phase 5.1 and are confirmed in the additional_context requirement IDs list.

**How to avoid:** Include ROUL-05 and ROUL-06 in the VERIFICATION.md requirements coverage. Evidence is present: RouletteTable.tsx renders `dozen_1/dozen_2/dozen_3` and `col_1/col_2/col_3` zones; rouletteService.ts handles all six in `isBetWon()`.
