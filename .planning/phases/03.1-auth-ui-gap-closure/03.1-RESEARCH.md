## Summary

This is a surgical two-file change. The v1.0 milestone audit identified one integration gap (AUTH-03) and one medium bug (session-expired banner unreachable). Both live entirely in the frontend presentation layer — no backend, no new dependencies, no schema changes.

**Gap 1 — Logout button (AUTH-03):** `signOut()` is fully implemented in `AuthContext.tsx` (lines 82–91): it calls `POST /api/auth/logout`, clears the module-scope access token via `setAccessToken(null)`, resets React state, and calls `clearBalance()`. The Header component does not import `useAuth` and renders no logout trigger. The fix is to import `useAuth`, destructure `signOut`, and add a button element with `onClick={signOut}`.

**Gap 2 — Session-expired banner unreachable:** `LoginPage.tsx` line 11 reads `params.get('expired') === 'true'` to decide whether to show the "Your session expired" banner. `ProtectedRoute.tsx` line 9 redirects to `"/login"` with no query string — so the banner is never displayed. The fix is one character change: `<Navigate to="/login?expired=true" replace />`. The `auth:session-expired` custom event fires from `client.ts` when a token refresh fails; `AuthContext.tsx` handles it by clearing state. When state clears, `accessToken` becomes null and `ProtectedRoute` re-renders, triggering the Navigate. With the param added, the redirect lands at `/login?expired=true` and the banner renders.

**Primary recommendation:** Edit `Header.tsx` (add logout button using `useAuth`) and `ProtectedRoute.tsx` (add `?expired=true` to the Navigate destination). No other files need to change.

### Pattern 1: Adding useAuth to Header.tsx

**What:** Import `useAuth` hook, destructure `signOut`, render a `<button>` that calls it.
**When to use:** Any component that needs to trigger logout.

**Current Header.tsx (lines 1-4):**
```tsx
import { Link } from 'react-router-dom';
import { BalanceDisplay } from './BalanceDisplay';

export function Header() {
```

**After modification:**
```tsx
// Source: existing AuthContext.tsx pattern — same hook used in LoginPage, RegisterPage
import { Link } from 'react-router-dom';
import { BalanceDisplay } from './BalanceDisplay';
import { useAuth } from '../contexts/AuthContext';

export function Header() {
  const { signOut } = useAuth();
```

**JSX addition** — the button goes inside the header flex container, after the BalanceDisplay link:
```tsx
<button
  onClick={() => void signOut()}
  style={{
    background: 'none',
    border: '1px solid rgba(255,255,255,0.3)',
    color: '#ffffff',
    cursor: 'pointer',
    padding: '6px 14px',
    borderRadius: '4px',
    fontSize: '0.875rem',
  }}
>
  Sign Out
</button>
```

Note: `signOut` is `async` (returns `Promise<void>`). Call via `onClick={() => void signOut()}` to avoid floating-promise lint warnings. TypeScript will flag `onClick={signOut}` as a type error in strict mode because React's onClick expects `void`, not `Promise<void>`.

### Pattern 2: Fixing ProtectedRoute.tsx session-expired redirect

**What:** Change the Navigate destination to include `?expired=true`.
**When to use:** Any redirect-to-login that should surface the session-expired banner.

**Current (line 9):**
```tsx
if (!accessToken) return <Navigate to="/login" replace />;
```

**After modification:**
```tsx
if (!accessToken) return <Navigate to="/login?expired=true" replace />;
```

**Concern to evaluate:** This change makes ALL unauthenticated redirects show the "Your session expired" banner — including first-time visitors who simply haven't logged in yet. Two options:

**Option A (simple, slightly wrong UX):** Use `?expired=true` unconditionally — a user who navigates to `/dashboard` without logging in will see "Your session expired". Technically incorrect but acceptable for v1.

**Option B (correct UX):** Track whether the user ever had a session. If `accessToken` was previously non-null and became null, redirect with `?expired=true`. If `accessToken` was always null (fresh visit), redirect to plain `/login`.

The audit fix description says to use `?expired=true` unconditionally (simple fix). The audit's stated scope is to make the banner reachable — not to perfect first-visit UX. Use Option A for this gap-closure phase.

### Anti-Patterns to Avoid

- **Putting signOut in DashboardPage instead of Header:** DashboardPage is not a layout component — once Phase 4 adds game pages, those pages won't have DashboardPage. Header is rendered inside DashboardPage now, but the audit notes this will need to become a global layout. Putting the logout button in Header is the correct placement.
- **Navigating after signOut in AuthContext:** `signOut()` should not call `navigate('/login')` itself — it is a pure auth action. Navigation is handled by the ProtectedRoute re-rendering when `accessToken` becomes null. Mixing navigation into signOut creates a dependency on router context inside AuthContext (fragile).
- **Using window.location.href for redirect:** React Router's `<Navigate>` component is already in place and handles the redirect correctly. Don't bypass it with imperative navigation.

## Common Pitfalls

### Pitfall 1: Floating Promise on onClick

**What goes wrong:** `onClick={signOut}` — TypeScript in strict mode flags this because React's `MouseEventHandler` returns `void`, not `Promise<void>`. The button may appear to work but generates a TS error.

**Why it happens:** `signOut` is declared `async` — it returns `Promise<void>`. React's event handler type is `() => void`.

**How to avoid:** Use `onClick={() => void signOut()}`. The `void` operator discards the promise return value and satisfies the type constraint.

**Warning signs:** TypeScript error: "Type '() => Promise<void>' is not assignable to type 'MouseEventHandler<HTMLButtonElement>'."

### Pitfall 2: Session-expired banner shows on every unauthenticated visit

**What goes wrong:** With `?expired=true` unconditionally in ProtectedRoute, a brand-new user navigating to `/` sees "Your session expired" — confusing message for someone who never logged in.

**Why it happens:** ProtectedRoute cannot distinguish "never had a session" from "had a session that expired."

**How to avoid:** Acceptable for v1 gap closure. Document as known UX limitation. Track for Phase 4 if needed.

**Warning signs:** User opens app fresh, sees session-expired message on login page.

### Pitfall 3: Header not yet a global layout

**What goes wrong:** Header is currently rendered only inside `DashboardPage`. The logout button will only appear on the dashboard. Phase 4 game pages will not show Header (and thus not show the logout button) unless they also render `<Header />`.

**Why it happens:** There is no shared layout wrapper in `App.tsx` — each protected page is responsible for rendering Header itself.

**How to avoid:** This phase does not need to fix the layout architecture. The audit notes it as tech debt. Document it and leave it for Phase 4 to address (the audit already flags it: "will silently break when Phase 4 adds game pages unless each new page includes `<Header />` or a shared layout wrapper is introduced before Phase 4 execution").

**Warning signs:** Game pages in Phase 4 missing balance display and logout button.

### Complete modified ProtectedRoute.tsx

```tsx
// Source: apps/frontend/src/components/ProtectedRoute.tsx (modified)
import { Navigate } from 'react-router-dom';
import { useAuth } from '../contexts/AuthContext';
import type { ReactNode } from 'react';

export function ProtectedRoute({ children }: { children: ReactNode }) {
  const { accessToken, isLoading } = useAuth();

  if (isLoading) return <div>Loading...</div>;
  if (!accessToken) return <Navigate to="/login?expired=true" replace />;
  return <>{children}</>;
}
```

### signOut function signature (AuthContext.tsx — read only)

const signOut = useCallback(async () => {

    // Ignore errors — clear local state regardless
  }
  setAccessToken(null);
  setState({ accessToken: null, isLoading: false });
